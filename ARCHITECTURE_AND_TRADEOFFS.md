# Architecture & Tradeoffs

### 1. Data Fetching & State Management
**How We Handled It**: 
Data fetches were strictly kept at the React Server Component (RSC) level. Whenever users alter pagination inputs, sorting, or complex search bar filters (such as searching through Intern lists or the IC Catalog), instead of relying on thick client-side `useState` loops or external dependencies like Redux, we pushed parameters natively to the URL (`?status=APPROVED_AS_NEW&page=1`). The server naturally picks these up and repopulates the DOM context for free, resulting in extremely lean client-side browser bundles. 

**Tradeoff**: 
While routing-based state natively syncs URL shareability, there is a minor lag associated with Server Actions and full-page repaints over instantaneous optimistic local updates, specifically for users with slower internet connections querying big sets directly.

### 2. Form Mutation & Atomic Validation
**How We Handled It**:
When intern additions are merged with current database records (Phase 5b naming resolutions), the `next/cache` function `revalidatePath` executes alongside Prisma backend commands via `use server`. Core queries are executed using `prisma.$transaction()`. 

**Tradeoff**: 
Writing complex backend actions inline with Next.js features (`Action` files) means backend logic and constraints (like Role requirements) are physically coupled to the Next server boundaries. In the future, if you wish to decouple the React components from the database to an external dedicated Go/Python microservice, most Action validation logic would have to be manually re-written into external REST endpoint guards.

### 3. User Authentication
**How We Handled It**:
For rapid MVP implementation and testing layouts seamlessly (Phase 2 constraint), we mocked out a persistent cookie interface `ESIM_USER_COOKIE` via the inline development `<DevUserSwitcher />` in the UI header.

**Tradeoff**:
You can switch natively testing UI branches between MENTORS and INTERNS instantly, but there is no OAuth context linking profiles to external Github/Google accounts securely yet. This will strictly require deployment re-configurations merging in standardized packages like NextAuth as production readies. 

### 4. Code Structuring & Third-Party Library Integration
**How We Handled It**:
Ranged native `shadcn/ui` component building directly into directories over bloat dependencies, and utilized raw React hooks (`useEffect`, `setTimeout`) to resolve dynamic action-triggers like debouncing input fields.

**Tradeoff**:
We ran into restrictive TypeScript typings due to Prisma schema ENUM bindings misaligning securely with standard HTML attributes generated by headless UI generic libraries inside Next 15 / React 19 node attributes. Rather than implement bloated external parsing layers (like `zod`), we forcefully cast schema states inline to keep rapid progress while maintaining strict terminal build capabilities (`npx tsc --noEmit` checks correctly validates`).
